## C. 发现的问题清单

以下均为建议优化项（🟡 / 🟢），无必改项。

### S1 — 待使用状态的预约过期后不会自动转换 🟡

| 维度 | 描述 |
|------|------|
| 涉及章节 | §6 状态设计全篇、§9.4 业务约束 |
| 问题 | 一条"待使用"预约在其 `EndTime` 过后，状态仍保持为"待使用"不变，**不会自动变更为任何过期状态**。这意味着：用户的"我的预约"列表会持续堆积过去的"待使用"记录（与即将到来的预约无法区分）；系统也无法区分"即将待使用"和"已过期未使用"。管理员需要手动逐条清理。虽然 MVP 功能不受影响（冲突检测和占闲计算用的是时间范围而非状态），但演示时看到大量过去的"待使用"记录对观感有影响。 |
| 建议 | 方法一（推荐）：在 `ReservationService` 的查询入口增加隐式过期逻辑——查询"待使用"记录时，如果 `EndTime < DateTime.Now`，自动视为"已过期"展示（不修改 DB）。方法二：在状态表中增加"已过期"状态，通过定时任务或懒清理转换。对于 MVP，方法一成本最低。 |

### S2 — P2 统计页"有效预约数"使用 CreatedAt 而非 StartTime 🟡

| 维度 | 描述 |
|------|------|
| 涉及章节 | §8.2 "有效预约数"行 |
| 问题 | 定义：`Status = '已完成' AND CreatedAt >= @today AND CreatedAt < @nextDay`，用途标注为"反映实际到馆量"。但"到馆"的时间维度应该是**预约的日期**（StartTime）而非**创建记录的日期**（CreatedAt）。举例：学生今晚 11 点预约了明天上午的座位，明天到馆后被管理员标记完成——此时 CreatedAt 是今天，StartTime 是明天，如果按 CreatedAt 统计，这笔"到馆"被计入了今天而非明天。 |
| 建议 | 将查询条件由 `CreatedAt >= @today` 改为 `StartTime >= @today AND StartTime < @nextDay`。同时保持 `CreatedAt` 索引不变（统计页另有今日预约总数指标使用 CreatedAt）。如果本意就是统计"今天创建的记录中完成了多少"，则建议修改用途说明，避免误导。 |

### S3 — AvailableSeats 计算可添加 MAX(0) 兜底 🟢

| 维度 | 描述 |
|------|------|
| 涉及章节 | §8.1 "AvailableSeats"行 |
| 问题 | `AvailableSeats = TotalSeats(IsEnabled=1) - 当前被占用的座位数(通过 Reservation 表计算)`。如果某次异常导致一个已禁用座位(`IsEnabled=false`) 存在一条有效预约，占用计数会包含该座位而总量不包含，导致可空闲数 < 0。虽然正常流程下不会发生（禁用座位不可预约），但防御性编程角度建议加 `MAX(0, ...)` 兜底。 |
| 建议 | Service 层计算时 `Math.Max(0, total - occupied)`。或者占用查询改为只统计启用座位上的占用：`JOIN Seats ON ... WHERE IsEnabled = 1`。 |
