# 图书馆座位预约系统 — 项目立项单

## 1. 项目基本信息

| 项目 | 内容 |
|------|------|
| 项目名称 | 图书馆座位预约系统 |
| 项目代号 | LibSeat |
| 技术栈 | ASP.NET Core MVC (.NET 8) + Razor + SQL Server LocalDB + EF Core + Bootstrap |
| 部署方式 | 本地 IIS Express / dotnet run 运行，浏览器访问 |
| 代码管理 | GitHub 或 Gitee 私有/公开仓库 |
| 开发工具 | Visual Studio 2022 + SQL Server (LocalDB) |
| 项目性质 | 课程第三阶段模拟企业项目 |

## 2. 项目背景

很多高校图书馆存在占座严重、座位利用率低的问题。学生一大早去图书馆发现桌上放着书但人不在，又不敢挪开；管理员也没办法实时掌握座位使用情况。

市面上有不少预约系统，但要么是商业软件不开放，要么功能太重（支付、扫码、多馆区联动）不适合校园小范围落地。本期项目就是做一个"够用、能跑、能演示"的轻量座位预约系统，让管理员能管座位，学生能约座位。

## 3. 目标用户

| 角色 | 描述 |
|------|------|
| 普通用户（学生） | 需要去图书馆自习的学生，想在出发前就知道有没有空位，到了直接坐 |
| 管理员（图书馆老师 / 勤工助学学生） | 需要管理座位开放情况，查看预约记录，统计使用数据 |

## 4. 项目目标

开发一个基于 ASP.NET Core MVC 的图书馆座位预约网站，满足以下核心诉求：

- **对学生**：能浏览座位、发起预约、取消预约、查看自己的预约记录
- **对管理员**：能管理座位、审核/处理预约、查看统计概览
- **对演示**：在本地能用浏览器打开，各页面可走通、可操作

## 5. 本期必须完成内容

以下为本期（MVP 阶段）必须交付的功能：

### 用户端（面向普通学生）

- [ ] **用户首页** — 展示公告、快捷入口、座位使用概览
- [ ] **座位列表** — 可按区域/楼层筛选座位，显示占用/空闲状态
- [ ] **座位详情** — 查看座位位置、设施、时段预约情况
- [ ] **预约提交** — 选择座位、选择预约时段、提交预约。时段按固定 slot 划分，每个 slot 1 小时（如 08:00-09:00, 09:00-10:00...），用户选择一个或多个连续 slot，时段范围与图书馆开放时间（08:00-22:00）绑定
- [ ] **我的预约** — 查看自己所有预约记录及状态
- [ ] **取消预约** — 对"待使用"的预约进行取消操作

### 管理端（面向管理员）

- [ ] **管理员登录** — 基于 Session/Cookie 的轻量登录。预设账号通过 EF Core Seed Data 初始化，默认用户名 `admin`，密码 `admin123`
- [ ] **预约管理** — 查看所有预约、筛选、手动结束/取消预约
- [ ] **座位管理** — 增删改座位（楼层、区域、座位号、启用/禁用）
- [ ] **统计页** — 展示 3 项最低指标：今日预约总数、当前有效预约数（待使用）、座位使用率（已预约 slot / 总 slot 百分比）。纯数字展示即可，无需图表

### 通用要求

- [ ] 数据通过 EF Core + SQL Server LocalDB 持久化
- [ ] 页面使用 Bootstrap 保证基本可用样式
- [ ] 用户端采用体验账号切换实现登录：提供 3~5 个预设体验用户（学生 A、学生 B、学生 C + 管理员），在切换页以卡片或下拉列表展示，点击即切换身份，无需密码

## 5a. 初步数据模型（共识级）

本期涉及两张核心表，立项阶段先约定主要字段，后续可微调：

### 座位表（Seat）

| 字段 | 类型 | 说明 |
|------|------|------|
| Id | int（主键） | 自增编号 |
| Floor | string | 楼层，如"2 楼"、"3 楼" |
| Area | string | 区域，如"A 区"、"自习区" |
| SeatNumber | string | 座位编号，如"A-01" |
| IsEnabled | bool | 是否启用，禁用后不可预约 |

### 预约表（Reservation）

| 字段 | 类型 | 说明 |
|------|------|------|
| Id | int（主键） | 自增编号 |
| SeatId | int（外键） | 关联的座位 |
| UserName | string | 预约人标识（对应体验账号用户名） |
| StartTime | datetime | 预约开始时间（如 2026-07-06 09:00） |
| EndTime | datetime | 预约结束时间（如 2026-07-06 10:00） |
| Status | string | 状态：待使用 / 已完成 / 已取消 |
| CreatedAt | datetime | 创建时间 |

## 6. 本期暂不开发内容

以下功能本期明确不做，作为后续扩展方向的参考：

- ❌ 支付 / 押金 / 罚款
- ❌ 扫码签到 / 闸机联动
- ❌ 消息推送（短信、邮件、App 推送）
- ❌ 真实账号注册 / OAuth 登录 / 密码找回
- ❌ 多馆区 / 多楼层联动（本期最多按楼层筛选）
- ❌ 黑名单 / 信用分
- ❌ 实时 websocket 座位状态推送（页面刷新即可）
- ❌ 移动端适配（本期 Bootstrap 响应式基本可用即可，不做独立 App 或小程序）

## 7. 页面规模与页面分布

共约 **12 ~ 14 个页面**，具体分布如下：

### 用户端（6 页）

| 页面 | 路由 | 说明 |
|------|------|------|
| 首页 | `/` | 公告、快捷入口、概览 |
| 座位列表 | `/Seats` | 楼层/区域筛选，状态展示 |
| 座位详情 | `/Seats/Detail/{id}` | 座位信息 + 时段预约情况 |
| 预约提交 | `/Reservation/Create` | 选座 + 选时段 + 提交 |
| 我的预约 | `/Reservation/My` | 个人预约列表 |
| 切换用户 | `/Account/Switch` | 展示预设体验用户列表（学生 A/B/C + 管理员），点击即切换身份 |

### 管理端（5 ~ 6 页）

| 页面 | 路由 | 说明 |
|------|------|------|
| 管理员登录 | `/Admin/Login` | Session 登录 |
| 管理首页 | `/Admin` | 快捷入口 + 概览 |
| 预约管理 | `/Admin/Reservations` | 查看/筛选所有预约 |
| 座位管理 | `/Admin/Seats` | 座位的增删改 |
| 统计页 | `/Admin/Statistics` | 数据统计展示 |

### 共享布局

- 用户端使用一套布局（带导航栏 + 底部）
- 管理端使用独立布局（侧边栏 + 顶部导航）

## 8. 关键业务主流程

### 流程一：学生预约座位

```
学生打开首页 → 点击"预约座位" → 浏览座位列表（筛选楼层/区域）
→ 点击某个座位查看详情 → 选择预约时段 → 点击"提交预约"
→ 系统检查时段是否冲突 → 成功则生成预约记录，状态为"待使用"
→ 跳转到"我的预约"可看到该记录
```

### 流程二：学生取消预约

```
学生打开"我的预约" → 找到状态为"待使用"的预约
→ 点击"取消预约" → 确认取消
→ 系统将预约状态改为"已取消" → 座位在该时段释放
```

### 流程三：管理员管理预约

```
管理员登录 → 进入"预约管理" → 查看所有预约列表
→ 可按状态（待使用/已完成/已取消）筛选
→ 手动将"待使用"标记为"已完成"（模拟签到）
→ 或强制取消某条预约
```

## 9. 成功标准

以下标准全部满足，即视为项目验收通过：

1. **首页可打开** — 运行项目后在浏览器访问 `http://localhost:xxxx` 能看到首页内容
2. **页面可导航** — 点击导航栏菜单能跳转到对应页面，不存在 404
3. **座位列表有数据** — 页面加载后能看到预设的座位数据（楼层、区域、状态）
4. **预约可提交** — 选座位、选时段、提交后能在"我的预约"中看到记录
5. **预约可取消** — 状态为"待使用"的预约可以取消，取消后座位状态恢复
6. **管理员可登录** — 输入 Seed Data 初始化的预设管理员账号（用户名 `admin`，密码 `admin123`）能进入管理后台
7. **管理员可管理座位** — 能新增、编辑、禁用座位
8. **统计页有数据** — 统计页至少显示 3 项指标：今日预约总数、当前有效预约数（待使用）、座位使用率（已预约 slot / 总 slot 百分比）。纯数字展示即可，无需图表
9. **数据库持久化** — 关闭程序重新启动后，已创建的预约数据不丢失
10. **代码提交到仓库** — GitHub/Gitee 仓库链接可访问，管理员能看到 commit 记录
11. **页面响应正常** — 本地运行环境下，页面切换和操作响应时间不超过 3 秒

## 10. 风险与假设

| 风险 | 可能性 | 影响 | 应对措施 |
|------|--------|------|----------|
| 开发时间不够，页面做不完 | 中 | 高 | 优先保核心流程（预约 + 取消 + 管理登录），统计页可以后期补 |
| 页面数量失控，越加越多 | 中 | 中 | 严格按本期清单开发，新增页面必须经过确认 |
| SQL Server LocalDB 环境配置不成功 | 中 | 高 | 准备好备选方案：改用 SQLite 作为兜底数据库 |
| EF Core 模型设计不合理导致改表麻烦 | 中 | 中 | 第一版先简化模型（不建过多关联表），后续再迭代 |
| AI 生成的代码不稳定，出现隐蔽 bug | 高 | 中 | 每完成一个功能手动走一遍主流程，及时发现并修复 |
| Bootstrap 样式调整耗时过长 | 高 | 低 | 本期不追求精美 UI，功能可用即可，样式只做基本对齐和配色 |
| Git 操作不熟练导致代码丢失或冲突 | 中 | 高 | 每次提交前确认状态，重要节点做备份；推荐使用 GitHub Desktop |

**关键假设**：

- 开发机器已安装 Visual Studio 2022 和 SQL Server LocalDB
- 开发机可联网，用于安装 NuGet 包和提交 Git
- 项目管理方接受 LocalDB 作为数据库方案（无需独立安装 SQL Server 实例）
