# 图书馆座位预约系统 — PRD Lite

> 前置文档：`docs/01-项目立项单.md`、`docs/02-需求分析与MVP确认.md`
> 本文件是轻量 PRD，直接服务于开发实现，不做过度设计

---

## 1. 项目概述

### 1.1 项目定位

一个面向高校图书馆的轻量座位预约 Web 系统，让学生可以在线查看座位余量、预约座位、取消预约；让管理员可以管理座位、查看预约记录。

### 1.2 技术选型

| 层 | 选型 |
|----|------|
| 后端框架 | ASP.NET Core MVC (.NET 8) |
| 视图引擎 | Razor |
| 数据库 | SQL Server LocalDB |
| ORM | Entity Framework Core |
| 前端框架 | Bootstrap 5 |
| 开发工具 | Visual Studio 2022 |

### 1.3 部署方式

本地运行（IIS Express 或 `dotnet run`），浏览器访问 `localhost` 端口。

---

## 2. 项目目标

1. **对学生**：在不安装任何 App 的情况下，用浏览器查看座位、提交预约、取消预约
2. **对管理员**：在浏览器中管理座位信息、查看和处理预约记录
3. **对开发者**：在 10 天周期内完成一个可演示、可操作、代码可提交的 MVP

---

## 3. 目标用户

| 角色 | 身份 | 核心诉求 | 技术能力 |
|------|------|---------|---------|
| 学生（普通用户） | 在校大学生，去图书馆自习 | 快速知道空位、预约、取消 | 会用浏览器 |
| 管理员 | 图书馆老师 / 勤工助学学生 | 管座位、看预约、查数据 | 会点菜单、看数字 |

## 4. 使用场景

### 场景 1：出门前看看有没有座

> 周四晚上，学生 A 打开系统，看到 3 楼自习区还有空位。他选了一个靠窗座位，预约明天 09:00-12:00。第二天到图书馆直接坐下。

### 场景 2：计划变了取消预约

> 学生 A 早上起晚了。打开"我的预约"，把那条"待使用"的预约取消，座位释放给别人。

### 场景 3：管理员处理预约

> 管理员 B 登录后台，看到今天预约列表，把已经来签到的学生预约标记为"已完成"，把没来的取消掉。

### 场景 4：管理员更新座位

> 图书馆新加了 10 个座位。管理员 B 登录后台逐个添加，启用后学生在座位列表中就能看到。

## 5. 用户端核心功能

### F-U01：切换用户

| 项目 | 内容 |
|------|------|
| 描述 | 用户通过点击预设体验账号来"登录"系统，无需密码 |
| 入口 | 首次访问自动跳转，之后通过导航栏"切换用户"进入 |
| 交互 | 页面展示 3~5 个预设用户卡片（学生 A、学生 B、学生 C、管理员），点击即切换，无需确认 |
| 行为 | 切换后自动跳转到座位列表页 |
| 数据影响 | 当前登录用户名写入 Session |

**最小验收标准：**
- [ ] 访问 `/Account/Switch` 能看到预设用户列表
- [ ] 点击任一用户卡片后，页面跳转到座位列表
- [ ] 之后再访问其他用户端页面，页面顶部显示当前登录的用户名

---

### F-U02：座位列表

| 项目 | 内容 |
|------|------|
| 描述 | 展示所有座位的概览信息，支持按楼层/区域筛选 |
| 入口 | 导航栏"座位列表" |
| 交互 | 页面以表格或卡片形式展示座位（编号、楼层、区域、状态）。顶部提供楼层和区域的下拉筛选框。点击座位行跳转到独立详情页 `/Seats/Detail/{id}`（位置描述 + 当前时段占用图） |
| 数据来源 | 从 Seat 表读取，结合 Reservation 表计算当前是否被占用 |

**最小验收标准：**
- [ ] 页面加载后能看到 10 条以上座位数据
- [ ] 每个座位显示：编号、楼层、区域、空闲/占用状态
- [ ] 切换楼层下拉框后，列表只显示对应楼层的座位
- [ ] 点击某座位能弹出或展开显示该座位详情

---

### F-U03：预约提交

| 项目 | 内容 |
|------|------|
| 描述 | 选择座位和时段 → 提交预约 → 系统检查冲突 → 生成预约记录 |
| 入口 | 在座位列表点击"预约"按钮，或通过独立页面 |
| 交互 | ① 选择座位（从座位列表选择）；② 选择预约日期；③ 选择时段（勾选一个或多个连续 1h slot，范围 08:00-22:00）；④ 点击"提交预约" |
| 冲突检测 | 提交时后端校验：同一座位同一时段不存在其他"待使用"或"已完成"的预约 |
| 反馈 | 成功 → 跳转到"我的预约"并显示成功消息；失败 → 在当前页提示冲突 |

**最小验收标准：**
- [ ] 能选择一个座位并选择一个时段后提交
- [ ] 提交成功后，在"我的预约"中能看到该记录，状态为"待使用"
- [ ] 同一座位同一时段重复提交时，页面提示冲突，不生成记录
- [ ] 提交时未选择时段或座位时，页面给出提示

---

### F-U04：我的预约

| 项目 | 内容 |
|------|------|
| 描述 | 当前用户所有预约记录列表 |
| 入口 | 导航栏"我的预约" |
| 交互 | 表格展示：座位编号、日期、时段、状态（待使用/已完成/已取消）、操作按钮。按创建时间倒序排列 |
| 操作 | 状态为"待使用"的每条记录右侧显示"取消预约"按钮 |

**最小验收标准：**
- [ ] 页面能显示当前用户的所有预约记录
- [ ] 每条记录显示完整信息：座位、时段、状态
- [ ] "待使用"的记录能看到"取消预约"按钮，"已完成"/"已取消"的记录没有该按钮

---

### F-U05：取消预约

| 项目 | 内容 |
|------|------|
| 描述 | 用户将"待使用"状态的预约取消 |
| 入口 | "我的预约"页面中点击"取消预约"按钮 |
| 交互 | 点击后弹窗确认 → 确认后状态改为"已取消" → 释放对应时段的座位 |
| 约束 | 仅"待使用"状态可取消；"已完成"和"已取消"不可操作 |

**最小验收标准：**
- [ ] 点击"取消预约"后弹确认框
- [ ] 确认后该预约状态变为"已取消"
- [ ] 取消后该座位该时段恢复空闲，其他用户可以预约

## 6. 管理端核心功能

### F-A01：管理员登录

| 项目 | 内容 |
|------|------|
| 描述 | 管理员通过预设账号登录管理后台 |
| 入口 | 访问 `/Admin/Login` |
| 交互 | 输入用户名 + 密码 → 提交 → 验证通过后写入 Session → 跳转到预约管理 |
| 账号 | Seed Data 预设：用户名 `admin`，密码 `admin123` |

**最小验收标准：**
- [ ] 访问 `/Admin/Login` 能看到登录表单
- [ ] 输入正确账号密码后能跳转到管理端页面
- [ ] 输入错误密码时页面提示"用户名或密码错误"
- [ ] 未登录直接访问管理端页面时，自动跳转到登录页

---

### F-A02：预约管理

| 项目 | 内容 |
|------|------|
| 描述 | 管理员查看所有预约记录，支持筛选和状态操作 |
| 入口 | 管理端导航栏"预约管理" |
| 交互 | 表格展示所有预约（预约人、座位、时段、状态、创建时间）。顶部提供下拉筛选（全部/待使用/已完成/已取消）。每条记录右侧提供操作按钮：待使用 → "标记完成"和"取消"；已完成/已取消 → 无操作 |
| 操作 | "标记完成" → 状态改为"已完成"；"取消" → 状态改为"已取消" |

**最小验收标准：**
- [ ] 页面能显示所有用户的预约记录
- [ ] 按状态筛选下拉能正确过滤记录
- [ ] 对"待使用"预约点"标记完成"后状态变为"已完成"
- [ ] 对"待使用"预约点"取消"后状态变为"已取消"

---

### F-A03：座位管理

| 项目 | 内容 |
|------|------|
| 描述 | 管理员对座位进行增、删、改、启用/禁用操作 |
| 入口 | 管理端导航栏"座位管理" |
| 交互 | 页面以表格展示所有座位（编号、楼层、区域、座位号、状态）。提供"新增座位"按钮。每条记录右侧提供"编辑"和"启用/禁用"按钮 |
| 新增 | 弹窗或跳转到表单页面：填写楼层、区域、座位号 → 提交后状态默认为启用 |
| 编辑 | 可修改楼层、区域、座位号 |
| 启用/禁用 | 切换 IsEnabled 状态；禁用后的座位在用户端座位列表中不可见或标记为不可预约 |

**最小验收标准：**
- [ ] 页面能显示所有座位
- [ ] 点击"新增"能添加座位，添加后出现在列表中
- [ ] 点击"编辑"能修改座位信息
- [ ] 点击"禁用"后，该座位在用户端不再显示为可预约状态
- [ ] 点击"启用"后恢复

## 7. 页面清单

共 **7 个功能页面 + 2 套布局 = 9 页口径**，与后续页面树阶段对齐。

> 说明：MVP 确认阶段计为 8 页（含 1 套共享布局），本 PRD 将用户端和管理端布局分开计数，功能页面数（7 页）不变。
> 
> 管理端不使用 Area 机制，视图文件直接放在 `Views/Admin/` 目录下，通过属性路由映射 URL。详见 `docs/07-系统设计说明.md` §6.2。

### 用户端（4 个功能页面 + 1 套布局）

| # | 页面 | 路由 | 功能清单 | 对应功能 |
|---|------|------|---------|---------|
| 1 | 用户共享布局 | `Views/Shared/_Layout.cshtml` | 导航栏（座位列表、我的预约、切换用户）+ 显示当前登录用户名 | 全局 |

### 管理端（3 个功能页面 + 1 套布局）

| # | 页面 | 路由 | 功能清单 | 对应功能 |
|---|------|------|---------|---------|
| 6 | 管理端布局 | `Views/Admin/Shared/_Layout.cshtml` | 侧边栏/顶栏导航（预约管理、座位管理）+ 登出 | 全局 |

```
# 管理端
/Admin/Login                    → 管理员登录
/Admin/Reservations             → 预约管理

```

## 11. 核心数据对象

### 11.1 座位表（Seat）

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| Id | int | PK, 自增 | 座位唯一编号 |
| Floor | string | 必填 | 楼层，如"2 楼"、"3 楼" |
| Area | string | 必填 | 区域，如"A 区"、"自习区" |
| SeatNumber | string | 必填, 楼层层内唯一 | 座位编号，如"A-01" |
| IsEnabled | bool | 默认 true | 是否启用；false 时前端不可预约 |

**初始 Seed Data**：覆盖 2~3 个楼层，共 15~20 个座位。

### 11.2 预约表（Reservation）

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| Id | int | PK, 自增 | 预约唯一编号 |
| SeatId | int | FK → Seat.Id | 关联座位 |
| UserName | string | 必填 | 预约人（对应体验账号用户名） |

## 12. 状态流转说明

### 12.1 预约状态图

```
          ┌─────────────────────────────────────┐
          │                                     │
          ▼                                     │
   ┌────────────┐    手动标记完成    ┌────────┐ │
   │  待使用     │ ───────────────→ │ 已完成  │ │
   │ (Pending)   │                  │ (Done)  │ │
   └─────┬──────┘                  └────────┘ │
         │                                    │
         │ 取消预约 (用户或管理员)              │
         ▼                                    │
   ┌────────────┐                             │
   │  已取消     │ ────────────────────────────┘
   │ (Cancelled) │
   └────────────┘
```

### 12.2 状态定义

| 状态 | 含义 | 触发动作 | 谁可触发 |
|------|------|---------|---------|
| 待使用 | 预约已提交，尚未到馆使用 | 预约提交成功 | 系统自动 |
| 已完成 | 已到馆使用或管理员标记完成 | 管理员手动标记"完成" | 管理员 |
| 已取消 | 预约被取消，座位释放 | 用户取消 / 管理员取消 | 用户、管理员 |

### 12.3 座位在预约列表中的状态

座位在座位列表中的"空闲/占用"判断逻辑：

> **按"当前时间"判断**：如果当前时间落在某个"待使用"或"已完成"预约的 [StartTime, EndTime) 区间内，则该座位显示为"占用"；否则为"空闲"。

## 15. 风险与假设

### 风险清单

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 开发时间不够，功能做不完 | 中 | 高 | 优先保障 P0 功能（预约+取消+管理登录），P2 功能可舍弃 |
| SQL Server LocalDB 环境配置失败 | 中 | 高 | 备选方案：改用 SQLite，EF Core 切换成本低 |
| EF Core 模型设计不合理导致改表 | 中 | 中 | MVP 阶段只建 3 张简单表，不建复杂关联 |
| AI 生成代码不稳定，出现隐蔽 Bug | 高 | 中 | 每完成一个功能手动走一遍主流程 |
| 页面数量失控 | 中 | 中 | 严格按 PRD 页面清单开发，新增页面需确认 |
| Git 操作失误导致代码丢失 | 中 | 高 | 重要节点备份；推荐 GitHub Desktop |

- 开发机已安装 Visual Studio 2022 和 SQL Server LocalDB

- 接受 LocalDB 作为数据库方案（无需独立安装 SQL Server）
- 所有体验账号和数据初始化通过 EF Core Seed Data 完成
- 演示时使用本地浏览器，不涉及外网部署
