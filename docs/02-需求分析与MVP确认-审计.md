# 需求分析与 MVP 确认 — 审计报告

> 审计对象：`docs/02-需求分析与MVP确认.md`
> 审计日期：2026-07-06
> 审计方法：逐项对照六维检查标准

---

## A. 总体判断

**结论：🟡 有条件通过，修复 1 项必改问题后即可进入 PRD Lite 阶段。**

本文档质量较高，结构完整，逻辑链条清晰。它不是立项单的简单改写，而是真正做了需求分析工作：定义了用户画像、提炼了核心场景、构建了候选功能池、进行了 4 层优先级分级、实施了 MVP 收紧决策、并产出了明确的删减/暂缓清单。

存在的主要问题是：**P1 层级的 A06（预约筛选）在 MVP 范围决策中未被明确处理**，导致"优先级 → MVP 选择"的决策链出现断裂。此外有若干可优化的表述问题。

---

## B. 发现的问题清单

### 问题 1 — A06（预约筛选）的 MVP 归属未闭合 🔴 必改

| 维度 | 描述 |
|------|------|
| 涉及章节 | §4 优先级矩阵 → P1 列表、§5 MVP 范围、§6 MVP 裁减项 |
| 问题 | **A06（预约筛选）在 §4 中被定为 P1（主体业务）**，但在 §5 MVP 包含列表中既未出现，在 §6 MVP 裁减项中也未提及。读者无法判断：A06 是包含在 A03（预约管理）中一起做了？还是被裁减了？如果是裁减了，为什么一个 P1 功能被裁减？ |
| 后果 | 进入 PRD Lite 后，开发方与评审方对"预约管理页面要不要筛选功能"的理解可能不一致，导致返工或争议。 |

### 问题 2 — "8 个页面"包含了共享布局，计数口径有歧义 🟡 建议优化

| 维度 | 描述 |
|------|------|
| 涉及章节 | §5 MVP 包含的页面 |
| 问题 | 列表中将"共享布局 `_Layout.cshtml`"计为 1 个页面，得出"共 8 个页面"的结论。但共享布局是所有页面共用的基础设施，不是独立页面。**实际功能页面为 7 个。** 如果不加说明，读者会认为有 8 个独立页面，到开发时会发现数不对。 |
| 后果 | 口径不一致会导致后续工作量估算偏差。 |

### 问题 3 — §7"保留项"与 §5 MVP 列表完全重复 🟡 建议优化

| 维度 | 描述 |
|------|------|
| 涉及章节 | §7 保留项 |
| 问题 | §7 的"保留项"表格（11 行）与 §5 MVP 包含的功能（11 行）基本一致，属于冗余信息。§7 的评审结论部分只需写"与 MVP 范围一致，无变更"即可，不需要重复粘贴。 |
| 后果 | 文档长度膨胀，后续维护时两处需同步更新，容易不一致。 |

### 问题 4 — 演示可看性 🟡 但缺少缓解措施 🟡 建议优化

| 维度 | 描述 |
|------|------|
| 涉及章节 | §7 评审维度 → 演示可看性 |
| 问题 | 评审正确指出了"MVP 没有首页和统计页，演示时可以先展示座位列表作为入口"，**但没有给出具体的演示流程建议**（比如：演示时浏览器直接打开 `/Seats`，然后按"预约→查看→取消→管理端"顺序走一遍）。缺少缓解措施意味着这个风险被识别但未管理。 |
| 后果 | 演示时可能手忙脚乱，不知道从哪开始。 |

### 问题 5 — §3 候选功能池缺少优先级标注 🟢 轻微

| 维度 | 描述 |
|------|------|
| 涉及章节 | §3 候选功能池 |
| 问题 | §3 列出了 18 项功能的完整清单（编号、名称、来源、描述），但没有标注每项的优先级（P0~P3）。读者看到清单时不知道哪些重要哪些次要，要到 §4 才能对照。 |
| 后果 | 阅读流畅性降低，但信息完整性不受影响。 |

## C. 必改项（进入 PRD Lite 前必须修复）

| 编号 | 问题 | 修改建议 |
|------|------|----------|
| **P1** | A06 的 MVP 归属未闭合 | 二选一：**① 如果 A06 的筛选能力已包含在 A03 MVP 范围内** → 在 §5 MVP 表中 A03 的"说明"列补充"含基本筛选（按状态）"，并在 §6 裁减项中说明"高级筛选（多条件组合）暂缓"；**② 如果 A06 被裁减** → 在 §6 MVP 裁减项中增加一行，说明"预约筛选功能暂缓，MVP 阶段预约管理仅展示全部列表"，并说明理由（如：数据量小，手动查找即可）。建议采用方案 ①，因为按状态筛选是预约管理的基本能力。 |

## D. 建议优化项（建议修复，但不阻塞）

| 编号 | 问题 | 优化建议 |
|------|------|----------|
| **S1** | "8 个页面"计数口径 | 将 §5 的"共 8 个页面"改为"**7 个功能页面 + 1 个共享布局**"，或将共享布局从页面计数中移除，在页面列表下方用注释放置布局说明。 |
| **S2** | §7 保留项冗余 | 将 §7 保留项表格替换为一句话："保留项与 §5 MVP 包含功能一致，无变更。" 原表格删除。 |
| **S3** | 演示可看性缺少缓解措施 | 在 §7 评审维度下方补充一段"演示建议"，例如："建议演示流程：① 打开 `/Seats` 展示座位列表 → ② 切换用户 → ③ 预约提交 → ④ 我的预约 → ⑤ 取消预约 → ⑥ 管理员登录 → ⑦ 预约管理 → ⑧ 座位管理。全程约 3~5 分钟。" |
| **S4** | §3 候选池加优先级列 | 在 §3 表格中增加"优先级"列，填入 P0~P3 标注，使读者在浏览候选池时就能感知功能重要性。 |

## E. 是否可以进入下一步"PRD Lite"

### 六维检查结果

| 维度 | 评价 |
|------|------|
| **1. 是否真的做了需求分析？** | ✅ **是**。本文档远超立项单范围：用户画像新增技术能力/使用频率维度（§1.1）、核心场景故事化（§1.2）、用户关注点排序（§2）、候选池按 ID 管理（§3）、4 层优先级矩阵（§4）、MVP 范围收束决策（§5）。是真正的需求分析工作，不是贴标签。 |
| **2. 是否有明确的功能优先级矩阵？** | ✅ **有**。4 层 P0~P3 定义清晰，每项功能都有"为什么是这个优先级"的说明，工作量占比参考合理。 |
| **3. 是否有明确的 MVP 范围确认？** | ✅ **有**。MVP = P0 + 选定 P1，有包含清单、页面列表、合并策略说明。 |
| **4. 是否有"删减项 / 暂缓项"？** | ✅ **有**。§6 分"立项排除项"和"MVP 裁减项"两类，§7 进一步分为"删减项"和"暂缓项"，决策理由清楚。 |
| **5. 需求评审结论是否成立？** | ⚠️ **基本成立，但 A06 未闭合（见 P1）。** 评审维度覆盖 4 个方面，结论合理。但 A06 的归属问题导致评审结论不完整。 |
| **6. 最终范围是否适合单人课堂项目？** | ✅ **适合**。从立项单的 12~14 页收紧到 7 个功能页面（+1 布局），2 张数据表，10 天 4 个里程碑。减负幅度合理（约 40%），保留了核心闭环。 |

### 综合裁定：🟡 有条件通过

修复 **P1（A06 MVP 归属闭合）** 后即可进入 PRD Lite 阶段。

S1~S4 四项优化建议中，**建议至少采纳 S1（页面计数口径）**，因为涉及工作量口头禅"8 个页面"可能误导后续规划。
