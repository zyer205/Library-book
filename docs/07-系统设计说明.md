## 1. 系统设计目标

### 1.1 本期目标

构建一个**可在本地运行、可演示、数据可持久化**的图书馆座位预约 MVP。不追求生产级架构，但要分层清晰，方便后续扩展。

### 1.2 设计约束

| 约束 | 原因 |
|------|------|
| 不引入额外 NuGet 包 | 新手环境配置越少越好，只用 .NET 8 内置 + EF Core + Bootstrap CDN |
| 不使用独立前端框架（Vue/React） | Razor 服务端渲染足以承载当前页面量，减少学习成本 |
| 不使用 Identity Framework | 无真实注册登录，预设账号切换，Session 存个用户名就够了 |
| 密码明文存储 | 仅限课堂演示，生产环境必须用 bcrypt |
| 不做分页 | 数据量小（<100 条），一个页面加载完 |
| 不使用 AJAX/API | 所有交互走表单 POST + 页面跳转，实习生好理解 |

### 3.2 目录约定

| 目录/文件 | 放什么 | 不放什么 |
|-----------|--------|---------|
| `Controllers/` | 只有路由、参数校验、调用 Service、返回 View 或 Redirect | 数据库查询、业务计算、ViewModel 拼装 |
| `Services/` | 业务逻辑：冲突检测、时段计算、状态流转 | 直接操作 `HttpContext`、返回 `IActionResult` |
| `Models/Entities/` | 实体类，属性直接对应数据库列 | 视图展示逻辑、验证注解（放在 ViewModel 层） |
| `Models/ViewModels/` | 一个 ViewModel 对应一个 View 的全部数据 | 实体引用、业务方法 |
| `Views/` | Razor 语法渲染 HTML，条件判断、循环展示 | 复杂 C# 计算、数据库访问 |
| `Program.cs` | 注册 Service、配置中间件、迁移 | 业务代码 |

> **判断依据**：一个文件是否职责单一？一个 Controller 方法是否有超过 15 行非路由处理代码？是的话，抽到 Service 去。

## 4. 分层说明

### 4.1 四层职责全景

完整的请求处理链路：

```
浏览器请求 → Controller → [ViewModel] → Service → [Entity] → DbContext → 数据库
                     ↓         ↓            ↓
                  接收参数   拼装数据    写业务逻辑
                  校验输入   传给View
                  返回结果
```

```
Controller 返回 View → Razor 渲染 → HTML 响应 → 浏览器展示
                              ↑
                         使用 ViewModel 数据
```

### 4.2 实体层（Entity，`Models/Entities/`）

**职责**：定义数据库表结构。每个 Entity 类对应一张表，属性对应列。不包含业务方法。

```csharp
// 典型写法（以 Seat.cs 为例）：
public class Seat
{
    public int Id { get; set; }
    public string Floor { get; set; } = "";
    public string Area { get; set; } = "";
    public string SeatNumber { get; set; } = "";
    public bool IsEnabled { get; set; } = true;

    // 导航属性（用于 EF Core 关联查询）
    public ICollection<Reservation> Reservations { get; set; } = new List<Reservation>();
}
```

**3 张表：**

| Entity | 表名 | 说明 |
|--------|------|------|
| `Seat` | Seats | 座位定义（编号、楼层、区域、是否启用） |
| `Reservation` | Reservations | 预约记录（关联 Seat、预约人、时间段、状态） |
| `Admin` | Admins | 管理员账号（用户名、密码明文） |

### 4.3 数据访问层（DbContext，`Models/AppDbContext.cs`）

**职责**：EF Core 上下文，负责 Entity 与数据库的映射、迁移、种子数据。

```csharp
public class AppDbContext : DbContext
{
    public AppDbContext(DbContextOptions<AppDbContext> options) : base(options) { }

    public DbSet<Seat> Seats => Set<Seat>();
    public DbSet<Reservation> Reservations => Set<Reservation>();
    public DbSet<Admin> Admins => Set<Admin>();

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        // 配置唯一约束、索引、默认值
        modelBuilder.Entity<Seat>(entity =>
        {
            entity.HasIndex(s => s.SeatNumber).IsUnique(); // 座位编号唯一
        });

        modelBuilder.Entity<Reservation>(entity =>
        {
            entity.Property(r => r.Status).HasDefaultValue("待使用");
            entity.Property(r => r.CreatedAt).HasDefaultValueSql("GETDATE()");
        });
    }
}
```

**关键约定：**
- 不要在 DbContext 之外写 SQL 或直接操作数据库
- 不要在 Controller 里 `new AppDbContext()`，通过依赖注入获取
- 迁移命令：`Add-Migration InitialCreate` → `Update-Database`

### 4.4 业务逻辑层（Service，`Services/`）

**职责**：所有"需要条件判断、计算、校验"的逻辑都放在这里。Controller 调用 Service，Service 调用 DbContext。

**需要的 Service：**

| Service 接口 | Service 实现 | 核心方法 |
|-------------|-------------|---------|
| `ISeatService` | `SeatService` | `GetAllAsync(filter)`、`GetByIdAsync(id)`、`CreateAsync(seat)`、`UpdateAsync(seat)`、`ToggleEnabledAsync(id)` |
| `IReservationService` | `ReservationService` | `CreateAsync(reservation)`——含冲突检测、`GetMyAsync(userName)`、`GetAllAsync(statusFilter)`、`CancelAsync(id)`、`MarkDoneAsync(id)` |

**重点方法的伪代码：**

```csharp
// ReservationService.CreateAsync 的核心逻辑
public async Task<(bool Success, string ErrorMessage)> CreateAsync(Reservation reservation)
{
    // 1. 校验时段合法性
    if (reservation.StartTime >= reservation.EndTime)
        return (false, "结束时间必须大于开始时间");

    // 2. 冲突检测：同一座位同一时段是否存在"待使用"或"已完成"的预约
    var conflict = await _context.Reservations.AnyAsync(r =>
        r.SeatId == reservation.SeatId &&
        r.Status != "已取消" &&
        r.StartTime < reservation.EndTime &&
        r.EndTime > reservation.StartTime);

    if (conflict)
        return (false, "该时段已被预约，请重新选择");

    // 3. 通过校验，保存
    _context.Reservations.Add(reservation);
    await _context.SaveChangesAsync();
    return (true, null);
}
```

**判断逻辑是否该进 Service 的标准：**

| 代码 | 该在哪 | 理由 |
|------|--------|------|
| `_context.Seats.Where(s => s.Floor == floor).ToListAsync()` | Service | 涉及数据库查询 |
| `r.Status != "已取消" && r.StartTime < endTime && r.EndTime > startTime` | Service | 业务条件判断 |
| `if (string.IsNullOrEmpty(userName)) return RedirectToAction("Index", "Home")` | Controller | 页面跳转是 Controller 职责 |
| `ModelState.AddModelError("", "冲突")` | Controller | 错误绑定到视图是 Controller 职责 |
| `seats.Count`、`reservations.Count` 等聚合 | Service | 数据加工 |
| `await _context.SaveChangesAsync()` | Service | 数据持久化 |

### 4.5 控制器层（Controller，`Controllers/`）

**职责：** 四件事——接参数、调 Service、拼 ViewModel、返回 View。

```csharp
// 标准写法（以 SeatsController 为例）：
public class SeatsController : Controller
{
    private readonly ISeatService _seatService;

    // 通过构造函数注入 Service
    public SeatsController(ISeatService seatService)
    {
        _seatService = seatService;
    }

    // GET: /Seats
    public async Task<IActionResult> Index(string floor = null)
    {
        var seats = await _seatService.GetAllAsync(floor);
        var viewModel = new SeatsIndexViewModel
        {
            Seats = seats,
            CurrentFloor = floor,
            Floors = await _seatService.GetDistinctFloorsAsync()
        };
        return View(viewModel);
    }

    // GET: /Seats/Detail/5
    public async Task<IActionResult> Detail(int id)
    {
        var seat = await _seatService.GetByIdAsync(id);
        if (seat == null) return NotFound();

        var viewModel = new SeatDetailViewModel
        {
            Seat = seat,
            // Service 计算当天时段占用数据
            TimeSlots = await _seatService.GetTimeSlotsAsync(id, DateTime.Today)
        };
        return View(viewModel);
    }
}
```

**Controller 的反面教材：**

```csharp
// ❌ 错误：Controller 直接操作数据库
public async Task<IActionResult> Index()
{
    var seats = await _context.Seats.ToListAsync(); // 不该在 Controller 里写 _context
    return View(seats);
}

// ✅ 正确：通过 Service 获取数据
public async Task<IActionResult> Index()
{
    var seats = await _seatService.GetAllAsync();
    return View(seats);
}
```

### 4.6 视图模型层（ViewModel，`Models/ViewModels/`）

**职责**：一个 ViewModel 类对应一个 View 页面，包含该页面需要的所有数据。不要直接传 Entity 到 View。

```csharp
// 示例：HomeIndexViewModel.cs
public class HomeIndexViewModel
{
    // 未登录态：展示预设用户列表
    public bool IsLoggedIn { get; set; }
    public List<UserCard> Users { get; set; }

    // 已登录态：展示概览仪表板
    public int TotalSeats { get; set; }
    public int AvailableSeats { get; set; }
    public int TodayReservations { get; set; }
    public string CurrentUserName { get; set; }
}

public class UserCard
{
    public string Name { get; set; }
    public string Role { get; set; }
    public string AvatarColor { get; set; }
}
```

**每一个 ViewModel 的内部字段正好对应 View 中 `@Model.xxx` 的每一次 `@Model` 引用。** 如果 View 中引用了 `@Model.xxx` 但 ViewModel 中没有这个属性，说明 ViewModel 定义不完整。

**View 用法示例：**

```html
@model SeatsIndexViewModel

<select>
    @foreach (var floor in Model.Floors)
    {
        <option value="@floor" selected="@(floor == Model.CurrentFloor)">@floor</option>
    }
</select>

@foreach (var seat in Model.Seats)
{
    <a asp-action="Detail" asp-route-id="@seat.Id" class="seat-card">
        <span>@seat.SeatNumber</span>
        <span class="badge-status @(seat.IsOccupied ? "occupied" : "available")">
            @(seat.IsOccupied ? "占用" : "空闲")
        </span>
    </a>
}
```

### 4.7 视图层（View，`Views/`）

**职责**：只做三件事——展示数据、循环列表、条件判断。不在 View 中写 `@functions`、不调 Service、不访问 `_context`。

**4.7.1 空状态处理（所有列表页都必须覆盖）**

每个列表页（座位列表、我的预约、预约管理、座位管理）在数据为空时必须展示空状态，不能显示空白页面。

```html
@if (Model.Seats.Count == 0)
{
    <div class="empty-state">
        <div class="empty-icon">🪑</div>
        <div class="empty-text">当前楼层暂无座位</div>
        <div class="empty-sub">试试切换其他楼层</div>
    </div>
}
else
{
    @foreach (var seat in Model.Seats)
    {
        // ... 正常列表
    }
}
```

涉及页面及空状态文案：

| 页面 | 空状态条件 | 文案 |
|------|-----------|------|
| 座位列表页 | 筛选后无结果 | "当前楼层暂无座位，试试切换其他楼层" |
| 我的预约页 | 当前用户无预约记录 | "你还没有预约记录，去预约一个座位吧" → 附带"去预约"按钮 |
| 预约管理页 | 无任何预约记录 或 筛选后无结果 | "暂无预约记录" |
| 座位管理页 | 无任何座位 | "暂无座位数据，请点击'新增座位'添加" → 附带新增按钮 |
| 统计页 | 无任何数据 | "暂无数据，请先添加座位和预约记录" |

**4.7.2 客户端表单校验（所有表单页都必须覆盖）**

所有表单页面（管理员登录、预约提交、新增/编辑座位）必须使用 Bootstrap 5 原生表单校验或 `required` 属性，在用户提交前阻止空表单。

```html
<!-- 在 form 上添加 novalidate 属性 + Bootstrap 校验 class -->
<form class="needs-validation" novalidate>
    <div class="mb-3">
        <label class="form-label">用户名</label>
        <input type="text" class="form-control" required>
        <div class="invalid-feedback">请输入用户名</div>
    </div>
    <div class="mb-3">
        <label class="form-label">密码</label>
        <input type="password" class="form-control" required>
        <div class="invalid-feedback">请输入密码</div>
    </div>
    <button type="submit" class="btn btn-primary">登录</button>
</form>

<script>
    // Bootstrap 5 表单校验激活脚本
    (function() {
        'use strict';
        var forms = document.querySelectorAll('.needs-validation');
        Array.prototype.slice.call(forms).forEach(function(form) {
            form.addEventListener('submit', function(event) {
                if (!form.checkValidity()) {
                    event.preventDefault();
                    event.stopPropagation();
                }
                form.classList.add('was-validated');
            }, false);
        });
    })();
</script>
```

涉及页面的必填字段：

| 页面 | 必填字段 | 错误提示 |
|------|---------|---------|
| 管理员登录 | 用户名、密码 | "请输入用户名" / "请输入密码" |
| 预约提交 | 时段选择（至少勾选一个 slot） | "请至少选择一个时段"（需自定义 JS 校验） |
| 新增/编辑座位 | 楼层、区域、座位号 | "请填写完整信息" |

**Layout 布局说明：**

| Layout | 位置 | 内容 |
|--------|------|------|
| 用户端 `_Layout.cshtml` | `Views/Shared/_Layout.cshtml` | 顶部导航栏（首页/座位列表/我的预约/用户名） + 底部 Tab 栏（手机端） + `@RenderBody()` |
| 管理端 `_Layout.cshtml` | `Views/Admin/Shared/_Layout.cshtml` | 左侧侧边栏（预约管理/座位管理/退出） + 顶部标题栏 + `@RenderBody()` |

**管理端手机端适配规则：**

管理端以桌面端为设计基准，手机端不做专门适配，但必须做到以下最低要求：

| 规则 | 实现方式 |

## 5. 模块划分

按功能域分为 4 个模块，每个模块包含 Controller + Service + ViewModel + View：

| 模块 | Controller | Service | ViewModel | View 文件 | 说明 |
|------|-----------|---------|-----------|----------|------|
| **首页模块** | `HomeController` | `ISeatService` + `IReservationService` | `HomeIndexViewModel` | `Home/Index.cshtml` | 展示身份选择或仪表板；**统计数字必须从 Service 查询真实数据**，不可在 View 中硬编码 |
| **座位模块** | `SeatsController` | `ISeatService` | `SeatsIndexViewModel`、`SeatDetailViewModel` | `Seats/Index.cshtml`、`Seats/Detail.cshtml` | 查询座位、计算占用状态 |
| **预约模块** | `ReservationController` | `IReservationService` | `ReservationCreateViewModel`、`MyReservationsViewModel` | `Reservation/Create.cshtml`、`Reservation/My.cshtml` | 预约 CRUD、冲突检测 |
| **管理端模块** | `Admin/LoginController`、`Admin/ReservationsController`、`Admin/SeatsController`、`Admin/StatisticsController` | `IReservationService`、`ISeatService`（复用） | 登录用 `LoginViewModel`；预约管理用独立的 `AdminReservationsViewModel`；座位管理用独立的 `AdminSeatsViewModel`；统计页用 `AdminStatisticsViewModel` | `Admin/Login.cshtml`、`Admin/Reservations.cshtml`、`Admin/Seats.cshtml`、`Admin/Statistics.cshtml` | 管理操作 |

### 6.2 路由注册（Program.cs）

管理端不使用 Area 机制。为了与 §3.1 的 **`Controllers/Admin/` 扁平目录**相匹配，管理端 Controller 使用属性路由（Attribute Routing），用户端使用默认路由：

**Program.cs 路由注册：**
```csharp
// 只有默认路由，不需要 area 路由
app.MapControllerRoute(
    name: "default",
    pattern: "{controller=Home}/{action=Index}/{id?}");
```

**管理端每个 Controller 顶部添加属性路由：**
```csharp
// Controllers/Admin/LoginController.cs
[Route("Admin/Login")]
public class LoginController : Controller { ... }

// Controllers/Admin/ReservationsController.cs
[Route("Admin/Reservations")]
public class ReservationsController : Controller { ... }

// Controllers/Admin/SeatsController.cs
[Route("Admin/Seats")]
public class SeatsController : Controller { ... }
```

> **为什么不用 Area？** Area 要求 Controller 放在 `Areas/Admin/Controllers/` 目录下，且需要 `[Area("Admin")]` 属性。如果按 §3.1 的 `Controllers/Admin/` 扁平目录放文件，Area 路由会找不到 Controller 从而导致 404。属性路由方案对实习生更直观——看到 `[Route("Admin/Login")]` 就知道 URL 是 `/Admin/Login`。
>
> ⚠️ **管理端 Controller 中的跳转需要注意**：由于使用属性路由，`RedirectToAction("Index", "Reservations")` 可能会产生 `/Reservations` 而非 `/Admin/Reservations`。建议在管理端 Controller 中统一使用 `Redirect("/Admin/Reservations")` 硬编码路径，或使用属性路由的 `[RouteName]` 命名路由方式跳转。

### 6.3 视图文件展开

```csharp
// Controllers/Admin/LoginController.cs
[Route("Admin/Login")]
public class LoginController : Controller
{
    [HttpGet]
    public IActionResult Index() => View("~/Views/Admin/Login.cshtml");
}
```

## 8. 异常处理原则

### 8.1 三层异常处理各管各的

| 层 | 处理什么异常 | 怎么做 |
|----|-------------|--------|
| Service | 业务异常（冲突、状态不合法、数据不存在） | 返回 `(bool Success, string ErrorMessage)` 元组，不抛异常 |
| Controller | 参数校验异常（ModelState 不合法） | `ModelState.AddModelError()` + 返回当前 View |
| Controller | Service 返回的失败结果 | 将 ErrorMessage 转为 `ModelState.AddModelError()` + 返回 View |
| 全局 | 未捕获异常（500 错误） | `app.UseExceptionHandler()` 指向通用错误页 |

### 8.2 具体做法

```csharp
// Service：返回结果元组，不抛异常
public async Task<(bool Success, string ErrorMessage)> CancelAsync(int id)
{
    var reservation = await _context.Reservations.FindAsync(id);
    if (reservation == null)
        return (false, "预约记录不存在");

    if (reservation.Status != "待使用")
        return (false, "只有'待使用'状态的预约可以取消");

    reservation.Status = "已取消";
    await _context.SaveChangesAsync();
    return (true, null);
}

// Controller：根据 Service 结果决定页面行为
[HttpPost]
public async Task<IActionResult> Cancel(int id)
{
    var result = await _reservationService.CancelAsync(id);
    if (!result.Success)
    {
        TempData["Error"] = result.ErrorMessage; // 在目标页面用 Alert 显示
    }
    return RedirectToAction(nameof(My));
}
```

### 8.3 不做什么

- 不在 Service 中写 `try-catch`（数据库异常由 EF Core 抛出，Controller 或全局中间件统一处理）
- 不在 View 中写异常处理
- 不抛出 `Exception` 类型（业务错误走返回值，不抛）

## 9. Session / Cookie 使用原则

### 9.1 本期用 Session 存什么

| Session Key | 类型 | 写入时机 | 清除时机 | 用途 |
|-------------|------|---------|---------|------|
| `UserName` | string | 用户切换身份时 | 切换用户 / 关闭浏览器 | 标识当前登录学生 |
| `IsAdmin` | bool | 管理员登录成功时 | 退出登录 / Session 超时 | 标识是否管理员身份 |

### 9.2 怎么用

```csharp
// 写入 Session
HttpContext.Session.SetString("UserName", "学生 A");
HttpContext.Session.SetInt32("IsAdmin", 1); // 用 int 表示 bool

// 读取 Session
string userName = HttpContext.Session.GetString("UserName");
bool isAdmin = HttpContext.Session.GetInt32("IsAdmin") == 1;

// 清除 Session（退出登录）
HttpContext.Session.Clear();
```

### 9.3 在 Controller 中的使用位置

| 场景 | 做法 |
|------|------|
| 需要检查是否已登录 | Action 方法顶部从 Session 读取 UserName |
| 需要限制仅管理员访问 | Action 方法顶部检查 IsAdmin，为 null 则 RedirectToAction("Index", "Login", new { area = "Admin" }) |
| 需要取当前预约人 | 创建 Reservation 时从 Session 取 UserName 赋值 |
| 需要传递给 View 显示用户名 | Controller 将 UserName 放入 ViewModel 或通过 `ViewBag.UserName` 传递 |

**不要在 View 中直接调用 `Context.Session`**。Controller 把值放到 ViewModel 或 ViewBag 中传给 View。

### 9.4 Session 配置（Program.cs）

```csharp
// Program.cs 中注册 Session 服务
builder.Services.AddDistributedMemoryCache();
builder.Services.AddSession(options =>
{
    options.IdleTimeout = TimeSpan.FromMinutes(30);
    options.Cookie.HttpOnly = true;
    options.Cookie.IsEssential = true;
});

// 中间件
app.UseSession();
```

### 9.5 Cookie

本期不需要主动写入或读取 Cookie。ASP.NET Core 的 Session 机制依赖一个 `.AspNetCore.Session` Cookie（自动管理），开发者不需要关心它。
