# 图书馆座位预约系统 — 需求分析与 MVP 确认

> 分析依据：`docs/01-项目立项单.md`
> 分析方法：课堂替代调研（校园场景假设 + 同类系统参考 + 课堂工作量约束）
> 本文件是 PRD 之前的收束动作，不是正式 PRD

---

## 1. 目标用户与核心场景

### 1.1 用户角色定义

| 角色 | 身份 | 技术能力 | 使用频率 | 核心诉求 |
|------|------|---------|---------|---------|
| 普通用户（学生） | 在校大学生，需要去图书馆自习 | 低，会用浏览器点链接 | 每天 1~2 次 | 快速知道有没有空位、预约、取消 |
| 管理员 | 图书馆老师或勤工助学学生 | 低，会点菜单看数据 | 每天 1 次 | 管座位、看预约、查统计 |

### 1.2 核心场景（课堂假设版）

**场景一：出门前看看有没有座**

> 周四晚上，学生 A 打算明天早上去图书馆自习。他打开系统，看到 3 楼自习区还有空位，选了一个靠窗座位，预约明天 09:00-12:00 三个小时。到图书馆直接坐下。

**场景二：计划变了取消预约**

> 学生 A 早上起晚了，来不及去图书馆。他打开"我的预约"，把那条"待使用"的预约取消掉，座位释放给其他人。

**场景三：管理员看看今天情况**

> 管理员 B 登录后台，看到今天有 23 条预约、12 个人正在使用座位。她顺便把 2 楼走廊边的几个座位设为"禁用"（因为那边在装修）。

**场景四：管理员更新座位布局**

> 图书馆新加了 10 个座位。管理员 B 登录后台，逐个添加新座位信息（楼层、区域、座位号），启用后学生就能在列表里看到了。

## 2. 用户最关心的问题

### 学生视角

| 排序 | 问题 | 重要性 |
|------|------|--------|
| 1 | **还有没有空座？** — 不用到图书馆就能知道 | 🔴 最高 |
| 2 | **怎么预约？** — 选座 + 选时间，越简单越好 | 🔴 最高 |
| 3 | **约了能不能取消？** — 计划有变时能主动释放 | 🔴 最高 |
| 4 | **我约了哪些？** — 查看自己的预约记录 | 🟡 高 |
| 5 | **座位在哪个位置？** — 能不能看到具体位置和环境 | 🟡 中 |

### 管理员视角

| 排序 | 问题 | 重要性 |
|------|------|--------|
| 1 | **怎么管理座位？** — 新增、禁用、修改座位信息 | 🔴 最高 |
| 2 | **谁约了哪个座？** — 查看所有预约记录 | 🔴 最高 |
| 3 | **座位使用率高不高？** — 有没有必要调整座位布局 | 🟡 中 |
| 4 | **操作要简单** — 不需要复杂的配置和培训 | 🟡 中 |

## 4. 功能优先级矩阵

### 分层标准

| 层级 | 标签 | 含义 | 课堂工作量参考 |
|------|------|------|--------------|
| **P0 核心基础** | 🏗️ 缺了不能跑 | 系统依赖的骨架功能，没有这些系统毫无意义 | 约 50% 工作量 |
| **P1 主体业务** | 🏠 有了才完整 | 完善系统价值的业务功能，没有会显得很空 | 约 30% 工作量 |
| **P2 体验提升** | ✨ 加分项 | 体验优化的锦上添花，没有也不影响功能跑通 | 约 15% 工作量 |
| **P3 暂缓** | ⏳ 以后再说 | 本期不做，留到后续迭代 | 0% 工作量 |

### 优先级矩阵

#### 🏗️ P0 核心基础

| 编号 | 功能 | 为什么是 P0 |
|------|------|------------|
| U02 | 座位列表 | 用户进来的第一屏，没有这个用户看不到任何信息 |
| U04 | 预约提交 | 核心业务操作，系统存在的首要理由 |
| U05 | 我的预约 | 用户需要知道自己约了哪些 |
| U06 | 取消预约 | 预约必须有释放机制 |
| U07 | 切换用户 | 没有真实注册，这就是用户的登录方式 |
| A01 | 管理员登录 | 管理端入口 |
| A04 | 座位管理 | 管理员必须能控制哪些座位可用 |
| G01 | 数据库持久化 | 刷新丢数据的话等于没用 |
| G04 | 预约冲突检测 | 同一座位同一时段不能约两次 |

#### 🏠 P1 主体业务

| 编号 | 功能 | 为什么是 P1 |
|------|------|------------|
| U03 | 座位详情 | 用户点了座位想看更多信息，但没有也不影响预约 |
| A03 | 预约管理 | 管理员需要查看和操作所有预约 |
| A06 | 预约筛选 | 预约多了需要筛选，减少翻页成本 |
| G03 | Seed Data 初始化 | 没有的话需要手动插数据，但数据库建好后再插也行 |

#### ✨ P2 体验提升

| 编号 | 功能 | 为什么是 P2 |
|------|------|------------|
| U01 | 用户首页 | 没有首页直接跳座位列表也能用 |
| U08 | 座位占用视图 | 让用户一眼看到哪些时段被占，但纯文字也能用 |
| A02 | 管理首页 | 没有就跳到预约管理也行 |
| A05 | 统计页 | 验收标准要求了，但放在最后做，不影响核心流程 |
| G02 | Bootstrap 样式 | 功能跑通后再美化，默认裸样式也能验收 |
| G05 | 404 页面 | 没有的话显示浏览器默认错误也行 |

#### ⏳ P3 暂缓

| 编号 | 功能 | 暂缓理由 |
|------|------|---------|
| — | 分页功能 | 课堂数据量很小（几十条），一页展示完全够 |
| — | 操作日志 | 谁什么时候做了什么，本期不需要追溯 |
| — | 邮件/短信提醒 | 立项已排除 |
| — | 扫码签到 | 立项已排除 |
| — | 批量导入座位 | 手动添加 10~20 个座位完全可接受 |
| — | 预约时间限制策略 | 如"提前 30 分钟才能预约"等规则，本期不做限制 |

## 5. MVP 范围确认

### MVP 定义

**MVP = P0（核心基础）+ P1（主体业务）中选定的必要项**

经过课堂工作量的收束分析，最终 MVP 范围如下：

### MVP 包含的功能

| 编号 | 功能 | 层级 | 说明 |
|------|------|------|------|
| **U02** | 座位列表 | P0 | 展示座位编号 + 楼层 + 区域 + 空闲/占用，简单筛选 |
| **U04** | 预约提交 | P0 | 选座位 → 选时段（1h slot）→ 提交 → 冲突检测 |
| **U05** | 我的预约 | P0 | 当前用户所有预约，显示状态 |
| **U06** | 取消预约 | P0 | 状态为"待使用"的预约可取消 |
| **U07** | 切换用户 | P0 | 预设用户列表，点击切换 |
| **A01** | 管理员登录 | P0 | Session 登录，admin/admin123 |
| **A03** | 预约管理 | P1 | 查看全部预约 + 手动标记完成/取消；含基本筛选（按状态：待使用/已完成/已取消） |
| **A04** | 座位管理 | P0 | 增删改座位 + 启用/禁用 |
| **G01** | 数据库持久化 | P0 | EF Core + LocalDB |
| **G03** | Seed Data 初始化 | P1 | 启动时初始化座位 + 管理员 |
| **G04** | 预约冲突检测 | P0 | 检查时段重叠 |

### MVP 包含的页面

共 **7 个功能页面 + 1 个共享布局**（比立项单少了 4~5 个独立页面，降低风险）：

| # | 页面 | 路由 | 归属于 |
|---|------|------|--------|
| 1 | 座位列表 | `/Seats` | 用户端 |
| 2 | 预约提交 | `/Reservation/Create` | 用户端 |
| 3 | 我的预约 | `/Reservation/My` | 用户端 |
| 4 | 切换用户 | `/Account/Switch` | 用户端 |
| 5 | 管理员登录 | `/Admin/Login` | 管理端 |
| 6 | 预约管理 | `/Admin/Reservations` | 管理端 |
| 7 | 座位管理 | `/Admin/Seats` | 管理端 |
| 8 | 共享布局 | `_Layout.cshtml` | 通用 |

> 座位详情页（U03）在 MVP 中合并到座位列表，点击座位行直接在列表内展开或跳转到一个简单详情视图。不单独作为一个完整页面开发。

### MVP 不包含但后续优先补的页面

| 页面 | 路由 | 计划阶段 |
|------|------|---------|
| 用户首页 | `/` | P2，如果时间够就做，否则直接重定向到座位列表 |
| 座位详情（独立页） | `/Seats/Detail/{id}` | P1，MVP 中合并到列表 |
| 管理首页 | `/Admin` | P2，重定向到预约管理即可 |
| 统计页 | `/Admin/Statistics` | P2，排在最后，不影响验收 |

## 6. 本期明确不做的功能

以下功能在本期（MVP 阶段）明确不做 —— 包含立项已排除项 + MVP 裁减项：

### 立项排除项（沿用）

| 功能 | 排除原因 |
|------|---------|
| 支付 / 押金 / 罚款 | 超出课堂范围 |
| 扫码签到 / 闸机联动 | 需要硬件，课堂无法实现 |
| 消息推送（短信、邮件） | 需要第三方服务 |
| 真实账号注册 / OAuth 登录 | 立项已确定为轻量模拟登录 |
| 多馆区联动 | 课堂只做一个馆区 |
| 黑名单 / 信用分 | 本期不需要规则引擎 |
| WebSocket 实时推送 | 页面刷新即可 |
| 独立 App / 小程序 | 只做 Web 端 |

### MVP 裁减项（本期立项有但按 MVP 节奏压后）

| 功能 | 原定 | MVP 决定 | 理由 |
|------|------|---------|------|
| 用户首页 | §5 必须完成 | **暂缓到 P2** | 没有首页，直接跳转座位列表不影响核心流程 |
| 座位详情独立页 | §5 必须完成 | **合并到座位列表** | 列表内展开详情或弹窗即可，不需要单独页面 |
| 统计页 | §5 必须完成 | **压到 P2 最后做** | 验收标准中有要求，但排在最后，如果时间不够可演示时口头说明 |
| 管理员首页 | §7 页面清单 | **暂缓到 P2** | 直接重定向到预约管理 |
| 楼层/区域高级筛选 | P1 预期 | **简化为下拉框** | 不做联动筛选、不做多条件组合 |
| 预约高级筛选（多条件组合） | A06 P1 | **合并到 A03 基本筛选** | MVP 阶段预约管理仅提供按状态（待使用/已完成/已取消）筛选，按日期/按用户等组合条件暂缓 |
| Bootstrap 精细样式 | §5 通用要求 | **裸 Bootstrap 即可** | 不做自定义 CSS 主题、不做响应式适配 |

## 7. 需求评审结论

### 评审维度

| 维度 | 结论 |
|------|------|
| 功能完整性 | ✅ MVP 覆盖了"预约 + 管理"闭环，所有用户故事可走通 |
| 课堂可行性 | ✅ 7 个功能页面 + 2 张数据表，10 天单人可完成 |
| 技术风险 | ⚠️ LocalDB + EF Core 对初学者有一定门槛，但项目骨架搭建后风险可控 |
| 演示可看性 | 🟡 MVP 没有首页和统计页，但核心流程完整，按顺序演示即可走通。**建议演示流程：** ① 打开 `/Seats` 展示座位列表 → ② 切换用户（选一个学生身份）→ ③ 预约提交 → ④ 我的预约确认记录 → ⑤ 取消预约 → ⑥ 管理员登录 → ⑦ 预约管理 → ⑧ 座位管理。全程约 3~5 分钟。 |

### 删减项（从本期范围移除）

## 8. 进入 PRD Lite 前的最终口径

### 一句话定义本次 MVP

> **一个"能约能管"的图书馆座位预约网站：学生切换体验账号 → 看座位 → 预约 → 取消；管理员登录 → 管座位 → 看预约。**

### MVP 关键数字

| 指标 | 数值 |
|------|------|
| 数据表 | 2 张（Seat + Reservation） |
| 页面数 | 7 个功能页面 + 1 个共享布局 |
| 用户端页面 | 4 个（座位列表、预约提交、我的预约、切换用户） |
| 管理端页面 | 3 个（登录、预约管理、座位管理） |
| 预设体验账号 | 3~5 个（学生 A/B/C + 管理员 admin） |
| 座位初始化 | 15~20 个（覆盖 2~3 个楼层/区域） |
| 开发周期 | 10 天 / 4 个里程碑 |
| 技术栈 | ASP.NET Core MVC .NET 8 + Razor + EF Core + LocalDB + Bootstrap |

### PRD Lite 需要明确的遗留问题

以下问题在本阶段尚未完全定论，需要在 PRD Lite 阶段拍板：

1. **座位列表中，占用状态的判断逻辑** — 是按"当前时间是否有正在进行的预约"还是按"当天是否有预约"？建议：按当前时间点判断，当前时间落在某个预约的 StartTime ~ EndTime 范围内即为"占用"。
2. **取消预约的时间窗口** — 是否可以随时取消？本期建议：不设限制，"待使用"状态均可取消。
3. **管理员手动结束预约的触发条件** — 是管理员在列表中点"标记完成"即可，还是需要输入备注？建议：直接点击标记完成，无需备注。
4. **切换用户后当前页面行为** — 切换后是留在当前页还是跳转到首页？建议：切换后自动跳转到座位列表。

> 以上 4 个问题在 PRD Lite 中以"设计决策记录"形式写入即可，不需要再做一轮调研。
