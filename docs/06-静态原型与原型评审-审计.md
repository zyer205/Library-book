## B. 发现的问题清单

### 问题 1 — slot 粒度：原型正确（1h），但规范 PC-03 仍写 30 分钟 🔵 注意

| 维度 | 描述 |
|------|------|
| 涉及对象 | `prototype/static-v1/seat-detail.html` vs `docs/05-页面卡与UI规范.md` PC-03 |
| 发现 | **上一轮审计的 P1 问题（slot 粒度不一致），原型作者在实践中选择了正确的方案**：seat-detail.html 的时段网格标签为 08:00、09:00、10:00……21:00，共 14 个 1 小时间隔，与 PRD 定义的 1h slot 一致。但规范文档 PC-03 仍写"每 30 分钟一格"。 |
| 影响 | **低风险，但文档落后于实现。** 需要在 `docs/05-页面卡与UI规范.md` 的 PC-03 中将"每 30 分钟一格"同步更新为"每 1 小时一格"，否则后续查看规范的人会困惑。 |

### 问题 2 — 用户端顶部栏出现"管理"入口，未在规范中定义 🔵 注意

| 维度 | 描述 |
|------|------|
| 涉及对象 | `index.html`、`seats.html`、`seat-detail.html`、`reservation-create.html`、`my-reservations.html`的顶部栏 |
| 发现 | **所有用户端页面的顶部栏右侧，在"切换"链接旁边出现了一个"管理"链接，指向 `admin/login.html`**。这个入口在页面树文档和 UI 规范中都没有定义。虽然对于单人课堂项目这是个方便的快捷方式（学生/管理员同一人测试），但它是一个未经讨论的设计决策。如果将来系统对普通用户开放，这个入口可能暴露管理端。 |
| 影响 | 如果本期是课堂项目且同一个人扮演所有角色，**可以接受，但建议在顶部栏加一个标注"仅在演示模式下显示"**，或加 HTML 注释说明。 |

### 问题 3 — 评审清单中 6 项修复缺少"进入开发前 vs 开发中"的分层 🟡 建议优化

| 维度 | 描述 |
|------|------|
| 涉及对象 | `prototype/review-1/原型评审清单.md` §9 |
| 发现 | §9 列出了 6 项修复事项，标注了 P2/P3 优先级。但没有进一步区分：哪些**必须在进入 MVC 开发前修复**（否则会增加后端改造工作量），哪些**可以在开发过程中逐步解决**。例如：9.1 座位列表空状态如果在开发前不补，意味着视图模板要 `if/else` 写两套，不如先补好空状态 HTML。 |
| 影响 | 开发者在进入 MVC 开发时不知道哪些修复需要做在前。 |

### 问题 4 — 部分内联样式与 CSS 变量体系不完全统一 🟢 轻微

| 维度 | 描述 |
|------|------|
| 涉及对象 | 用户端各 HTML 文件 |
| 发现 | CSS 中已定义 `--color-text-secondary: #64748B`、`--font-size-sm: 0.875rem` 等设计令牌，但多个页面中仍有 `style="font-size:12px;color:#64748B;text-decoration:none;"` 这样的内联写法。虽然静态原型中不影响，但**如果直接转换为 `.cshtml`，这些内联样式会增加 Razor 视图的维护成本**。 |
| 影响 | 转换到 MVC 时，建议将内联样式统一替换为 CSS 类。 |

## D. 建议优化项

| 编号 | 问题 | 优化建议 |
|------|------|----------|
| **S1** | slot 粒度文档落后于原型 | 将 `docs/05-页面卡与UI规范.md` PC-03 中的"每 30 分钟一格"改为"**每 1 小时一格（08:00-22:00，共 14 格）**" |
| **S2** | "管理"入口未定义 | 在顶部栏 HTML 中给"管理"链接加 `title="演示模式快捷入口"` 或注释 `<!-- 演示用，正式环境可移除 -->`。如果确认保留，在 UI 规范中补一条说明。 |
