## C. 必改项（进入 Sprint 0 前必须修复）

### P1 — 当前占用计算条件与数据库设计 §6.4 不一致 🔴

| 维度 | 描述 |
|------|------|
| 涉及位置 | §2.4.1 `GetAvailableCountAsync()` 第 409 行、§2.5 `IsOccupiedNowAsync()` 第 477 行 |
| 不一致内容 | **doc 09 使用 `Status == "待使用"`，doc 08 §6.4 SQL 使用 `Status != '已取消'`** |
| 具体代码 | **doc 09（本文件）：** `.Where(r => r.Status == "待使用" && r.StartTime <= now && r.EndTime > now)` |
| | **doc 08（数据库设计 §6.4）：** `WHERE Status != '已取消' AND StartTime <= @currentTime AND EndTime > @currentTime` |
| 差异影响 | 管理员标记一条预约"已完成"后：<br>- doc 09 做法：因为 `Status != "待使用"`，座位立即标记为"空闲" ✅<br>- doc 08 SQL 做法：因为 `Status != '已取消'`（已完成 ≠ 已取消），座位仍算"占用"直到 EndTime ❌ |
| 深层问题 | **doc 08 §6.1 状态表说"已完成→座位释放"，但 §6.4 SQL 却说"已完成仍算占用"——doc 08 内部也存在矛盾。** doc 09 用的 `Status == "待使用"` 在业务上更正确（管理员标记完成后座位应释放） |
| **修改建议** | **统一为 doc 09 的做法（`Status == "待使用"`），同时修正 doc 08 §6.4 的 SQL 示例**：<br>1. 本文件无需修改（逻辑正确）<br>2. 修改 doc 08 §6.4 SQL 中的 `Status != '已取消'` 为 `Status = '待使用'`，同时更新 §6.4 上方的说明文字为："当前时间 `t` 落在某条**待使用**预约的 `[StartTime, EndTime)` 区间内 → 占用"<br><br>**注意**：`GetTimeSlotsAsync()`（§2.4.1 第 429 行）使用的 `Status != "已取消"` 是正确的——时段占用图需要显示所有已预约的 slot（含已完成的），不应与当前占闲计算混用。两个查询目的不同，条件不同是合理的。 |

### P2 — 编辑座位（Edit）缺少 GET 路由和伪代码 🔴

| 维度 | 描述 |
|------|------|
| 涉及位置 | §3.2 路由表（只有 POST）、§3.3.3 Controller 动作（无伪代码） |
| 问题 | Edit 是管理端座位管理的标准 CRUD 操作。当前文档：<br>1. 路由表只有 `POST /Admin/Seats/Edit/{id}`，缺少 `GET` 路由（加载现有数据供编辑）<br>2. Controller 动作表列出了 `Edit(int id, SeatEditViewModel model)` 但没有伪代码实现<br>3. 对比之下 Create 和 Toggle 都有完整伪代码，Edit 是唯一的遗漏 |
| 建议修改 | 在 §3.2 路由表中补充 GET 路由：`\| 6a \| 编辑座位（GET）\| \`/Admin/Seats/Edit/{id}\` \| GET \| 点击"编辑"按钮，加载现有数据展示编辑表单 \|`。并在 §3.3.3 中为 Edit 补充 GET 和 POST 伪代码。伪代码可简短（因为逻辑与 Create 类似但多了预加载）：<br><br>```csharp<br>// GET: 加载现有数据<br>public async Task<IActionResult> Edit(int id) {<br>    var seat = await _seatService.GetByIdAsync(id);<br>    if (seat == null) return NotFound();<br>    var vm = new SeatEditViewModel { Id = seat.Id, Floor = seat.Floor, Area = seat.Area, SeatNumber = seat.SeatNumber };<br>    return View(vm); // 或作为 Index 页的局部视图<br>}<br><br>// POST: 保存修改<br>[HttpPost]<br>public async Task<IActionResult> Edit(int id, SeatEditViewModel model) {<br>    if (!IsAdmin()) return RedirectToAction("Index", "Login");<br>    if (!ModelState.IsValid) return View(model);<br>    var result = await _seatService.UpdateAsync(new Seat { Id = id, ... });<br>    if (!result.Success) TempData["Error"] = result.ErrorMessage;<br>    return RedirectToAction("Index");<br>}<br>``` |

## D. 建议优化项

### S1 — 每座位逐条 `IsOccupiedNowAsync` 导致 N+1 查询 🟡

| 维度 | 描述 |
|------|------|
| 位置 | §2.3.3 座位列表 → `foreach` 循环逐条调用 |
| 问题 | 当前实现：`foreach (var seat in seats) { seat.IsOccupied = await _seatService.IsOccupiedNowAsync(seat.Id); }`。对于 20 个座位需要 20 次独立数据库查询。在 LocalDB 本地环境下性能可接受（< 50ms），但代码层面存在优化空间。 |
| 建议 | 在 `SeatService` 中新增批量方法 `GetOccupiedSeatIdsAsync()`：一次性查询 `SELECT DISTINCT SeatId FROM Reservations WHERE Status = '待使用' AND StartTime <= @now AND EndTime > @now`，将结果返回为一个 `HashSet<int>`，然后在内存中赋值。这样一次查询替代 N 次。可在 Sprint 0 之后优化。 |

### S2 — 管理端新增座位输入校验失败使用 Redirect 导致数据丢失 🟡

| 维度 | 描述 |
|------|------|
| 位置 | §3.3.3 `Create` 第 786 行 |
| 问题 | `if (!ModelState.IsValid) return RedirectToAction("Index");`——Redirect 会丢失用户已填写的所有输入（Floor、Area、SeatNumber），用户必须重新填写。正确做法是返回当前 View（保留输入）并显示校验错误。 |
| 建议 | 如果新增座位表单是独立 View，改为 `return View(model)`（保留输入 + ModelState 错误）。如果表单是 Index 页内嵌的局部视图，则至少通过 `TempData` 回传用户输入。 |

### S3 — 未选身份时 `/Reservation/Create` 重定向到首页可能造成困惑 🟢

| 维度 | 描述 |
|------|------|
| 位置 | §2.3.5 第 238 行 |
| 问题 | 未选身份直接访问预约页时，重定向到首页（身份选择模式）。用户选完身份后需要重新导航到座位列表→详情→预约，而不是直接回到预约页。 |
| 建议 | 重定向到首页时保留 `returnUrl`：`return RedirectToAction("Index", "Home", new { returnUrl = $"/Reservation/Create?seatId={seatId}" })`。选完身份后跳转回 `returnUrl`。非必改，因为流程仍然走得通（只是多几步点击）。 |
