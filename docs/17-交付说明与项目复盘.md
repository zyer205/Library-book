## 三、做得好的地方

1. **架构清晰**：Controller → Service → DbContext 三层分离；接口与实现分离（IReservationService / ReservationService）。
2. **测试覆盖全面**：从手动 HTTP 测试 → 13 个 TC 用例 → Playwright 自动点击 23 用例 → 脚本烟雾测试 12 项 → 兼容性测试（移动端 + 桌面端）。
3. **命名规范一致**：`string?` 参数注解、`IsEnabled` 统一命名、`TempData` 反馈模式统一。
4. **性能优化**：N+1 查询优化（21 次 → 2 次），`HashSet.Contains()` 替代逐座循环。
5. **审计闭环**：每个 Sprint 都有对应的审计文档，发现的问题逐轮跟踪清零。
6. **Session 超时处理**：管理端和用户端都有对应的超时提示和重定向。
7. **已停用座位防护**：Service 层和 Controller 层双重检查，POST/GET 均安全。

## 六、最终演示顺序建议

建议按以下链路依次演示，总时长约 8-12 分钟：

### 链路一：用户端主链路（3-4 分钟）

```
1. 首页 → 展示 3 张身份卡片、系统名称
2. 点击"学生 A" → 切换到座位列表
3. 座位列表 → 展示 20 个座位、占闲状态、楼层筛选
4. 点击某可用座位 → 座位详情（时段占用图）
5. 点击"预约此座位" → 勾选时段 → 提交
6. 跳转到"我的预约" → 展示新预约记录
7. 点击"取消预约" → 确认 → 状态变"已取消"
```

### 链路二：管理端主链路（3-4 分钟）

```
1. 访问 /Admin/Login → 展示登录表单
2. 输入 admin / admin123 → 跳转到预约管理
3. 预约管理 → 状态筛选（全部/待使用/已完成/已取消）
4. 标记某条"待使用"为"已完成"
5. 进入座位管理 → 展示全部座位表格
6. 点击某座位的"禁用"按钮 → 确认
7. 切换到用户端 → 检查该座位已不在列表中
8. 回到管理端 → 重新启用该座位
9. 查看统计页（4 项 KPI）
```

### 链路三：异常处理（1-2 分钟）

```
1. 未登录直接访问 /Admin/Reservations → 跳转到 /Admin/Login?timeout=1
2. 显示"会话已超时，请重新登录"红色提示
3. 预约已停用的座位 → 显示"该座位已停用，无法预约"
```

### 链路四：自动化测试展示（1-2 分钟，可选）

```
1. 终端运行 npm test → 展示 23 个 Playwright 测试通过
2. 或运行 npm run smoke → 展示 12 项烟雾测试通过
```
